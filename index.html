📚 Глоссарий терминов

Простым языком · с аналогиями · для всех кто работает с отделом разработки
🎯 Триаж (Triage)
Первичная сортировка входящих задач, чтобы решить что делать дальше: отклонить, в беклог, в спринт или срочно в работу. Tech Lead делает за 48 часов после получения задачи.
Пример: пришла заявка → TL смотрит → решает «P2, в беклог» → задача ждёт груминга в среду
📋 Беклог (Backlog)
Список ВСЕХ задач которые надо сделать когда-нибудь, отсортированный по важности. Сверху — что важнее, снизу — что подождёт. Из беклога берём задачи в спринт.
Беклог: 1. Кнопка «Войти» 2. Фильтр предметов 3. Цвет фона 4. Тёмная тема
🏃 Спринт (Sprint)
Период времени (1–2 недели) в течение которого команда обещает сделать конкретный набор задач из беклога. У нас — 1 неделя. Пн планируем, Пт демо.
Спринт #3 (15-21.06) · Цель: учитель может создать класс и пригласить учеников
📋 Груминг (Grooming / Backlog Refinement)
Встреча в среду (1 час), где Tech Lead с заказчиками разбирает беклог: новые задачи, приоритеты, удаление дублей.
Среда 16:00: смотрим 5 новых заявок → обсуждаем → 2 в спринт, 2 в беклог, 1 отклоняем
🔥 P0 / P1 / P2 / P3 (Priority levels)
Система приоритетов задач. P0 — критично сейчас. P1 — важно сегодня. P2 — в этом месяце. P3 — когда время будет.
P0: сайт упал у всех → 4 часа. P1: оплата не работает → 24 часа. P2: дашборд тормозит → след. спринт. P3: кнопка кривая → беклог
📝 ТЗ (Spec / Technical Specification)
Документ описывающий: что нужно сделать, зачем, для кого, какой результат ожидаем. Без ТЗ задача в работу не идёт.
ТЗ содержит: цель, scope, user stories, макеты Figma, Definition of Done, оценку часов
✅ Definition of Done (DoD)
Чёткие критерии «когда задача считается завершённой». Защищает от ситуации «ну вроде сделал, но...».
DoD: код написан + ревью пройдено + тесты есть + работает на проде + заказчик принял
🌐 Staging / Production (Стейджинг / Прод)
Staging — копия боевого сервера для тестирования. Production — боевой сервер где работают реальные пользователи.
Сначала фичу выкатили на staging, заказчик потестил, принял → потом на production к реальным юзерам
🔀 Pull Request (PR / Merge Request)
Запрос разработчика на добавление его кода в общую кодовую базу. Сначала проходит code review, потом merge.
Разработчик сделал PR #42 → TL проверил → 2 замечания → исправил → approve → merge в main
🌿 Branch (Ветка / feature/XXX)
Отдельная «веточка» кода для одной задачи. Разработчик не пишет код прямо в main, а делает свою ветку.
feature/student-homework-flow — ветка для разработки сдачи ДЗ. После merge она удаляется.
👀 Code Review (Ревью кода)
Проверка кода другим разработчиком (обычно Tech Lead) перед попаданием в основную базу. Ищем баги, проблемы безопасности, нарушения стандартов.
PR создан → TL читает → пишет 5 комментариев → разработчик правит → re-review → approve
🔥 Hotfix (Экстренный фикс)
Экстренный фикс прямо в production, минуя обычный flow. Только для P0. Ветка от main, не от dev.
Прод упал в 14:00 → hotfix branch в 14:15 → fix в 15:30 → deploy в 16:00 → post-mortem завтра
🚀 Deploy / Release (Деплой / Релиз)
Процесс выкатки кода на staging или production. После deploy пользователи видят изменения.
Deploy в окно вт-чт 10:00-16:00. Никаких релизов в пятницу после 14:00.
↩️ Rollback (Откат)
Возврат к предыдущей версии если новый deploy сломал что-то. Делается за 5 минут.
Задеплоили в 15:00 → в 15:10 пошли ошибки → rollback в 15:15 → разбираемся почему сломалось
🔄 CI/CD (Continuous Integration / Continuous Deployment)
CI — автоматическая проверка кода (тесты, линтер) при каждом PR. CD — автоматический деплой после merge.
Push в ветку → GitHub Actions запускает тесты → merge → auto-deploy на staging
🧪 QA (Quality Assurance)
Контроль качества — тестирование что всё работает как задумано до того как попадёт к юзерам.
Разработчик закончил → QA берёт на staging → проходит тест-кейсы → находит 3 бага → возврат на правки
🐛 Bug / Issue (Баг / Задача)
Bug — что-то сломалось, работает не как должно. Issue — общее название для любой задачи в трекере.
Bug #142: кнопка «Сохранить» не работает в Safari. Issue #143: добавить новую секцию на дашборд.
💨 Smoke test (Дымовой тест)
Базовая быстрая проверка после деплоя — не упало ли что-то фундаментальное (логин, главная, основные функции).
После деплоя: открыть сайт, залогиниться, открыть урок, сдать ДЗ. Если работает — можно идти
🩺 Post-mortem (Разбор инцидента)
Документ-разбор после серьёзного инцидента (P0): что случилось, почему, как нашли, что починили, как избежать. Без обвинений.
Прод лежал 30 минут → post-mortem: причина в миграции БД → решение: добавить тест на это в CI
☕ Daily Standup (Дейли / Стендап)
15-минутная утренняя встреча команды. Каждый говорит: что сделал вчера / что делает сегодня / есть ли блокеры.
10:00 Zoom: каждый по 2 минуты. Если кто-то завис на проблеме — отдельный созвон после.
🎯 Sprint Planning (Планирование спринта)
Встреча в понедельник где команда выбирает задачи из беклога на новую неделю и оценивает их.
Пн 10:00: смотрим топ-10 беклога → берём 6 задач → распределяем → оцениваем часы
🎉 Retro (Ретроспектива)
Встреча в конце спринта (пт) где обсуждаем: что пошло хорошо, что плохо, что улучшить в процессе.
Пт 17:00: 3 хорошо (быстро задеплоили, помогли друг другу), 2 плохо (поздно созвонились, не успели тесты)
🛑 Blocker (Блокер)
Что-то что мешает разработчику работать. На дейли каждый говорит о блокерах, чтобы Tech Lead их разруливал.
Блокер: «жду доступ к staging серверу» → TL пишет DevOps → доступ за час → блокер снят
🛠️ Tech Debt (Технический долг)
Накопленные «костыли» в коде которые надо переписать когда будет время. Делаешь быстро сейчас — платишь временем потом.
Старый Express бэкенд который надо переписать на Postgres — это tech debt на 3 года
📐 Multi-tenancy (Мульти-тенантность)
Архитектура где одна система обслуживает много изолированных клиентов (школ). Каждая школа видит только свои данные.
Школа A не видит данных школы B. Реализуется через RLS политики Postgres
🔐 RBAC (Role-Based Access Control)
Система прав по ролям: учитель видит одно, ученик — другое, админ — всё. У каждой роли своя матрица разрешений.
Учитель: создавать уроки. Ученик: смотреть и сдавать ДЗ. Админ: всё + управление школой
📈 SLA (Service Level Agreement)
Соглашение о сроках. Например «P0 баги фиксим за 4 часа» — это SLA. Если нарушаем — претензии оправданы.
SLA: P0 < 4ч, P1 < 24ч, Новый функционал 2-6 нед, code review < 24ч
🏗️ MVP (Minimum Viable Product)
Минимально жизнеспособный продукт — самая базовая версия с которой уже можно запускаться и получать обратную связь.
MVP Амансултан: школа + учитель + урок + ДЗ. Без AI, без gamification, без аналитики глубокой
📞 War room (Военная комната)
Экстренный созвон команды когда P0 случился. Все бросают свои задачи, садятся вместе разруливать.

Готовые шаблоны в Google Docs · копируешь под свою задачу · заполняешь

📝 Шаблон ТЗ на разработку

Полный шаблон технического задания для любой новой фичи. Заполняется заказчиком вместе с Tech Lead до старта разработки.
Когда использовать: на этапе «Написание ТЗ» в процессе «Новый функционал». Без согласованного ТЗ задача в работу не идёт.
📂 Открыть в Google Docs 📋 Сделать копию

Структура документа (12 разделов):

  • Основная информация (название, проект, приоритет, исполнитель, дедлайн)
  • Контекст и бизнес-цель (зачем это нужно)
  • Описание задачи и User Stories
  • Scope — что входит и что НЕ входит
  • Функциональные требования (Frontend, Backend, интеграции)
  • Нефункциональные требования (производительность, безопасность)
  • Макеты и дизайн (Figma, брендбук)
  • Критерии приёмки (Definition of Done)
  • Зависимости и риски
  • Оценка и планирование (часы по ролям)
  • Дополнительные материалы
  • Согласования и история изменений

💰 Шаблон Акта приёмки работы разработчика

Документ для ежемесячной приёмки работы и расчёта оплаты разработчиков. Tech Lead проверяет, PM согласовывает, Owner утверждает.
Когда использовать: в конце каждого месяца (или по итогам этапа) для каждого разработчика отдельно. Без подписанного акта выплата не производится.
📂 Открыть в Google Docs 📋 Сделать копию

Структура документа (8 разделов):

  • Основная информация (номер, период, разработчик, ставка, приёмщики)
  • Выполненные задачи (таблица со ссылками на PR и статусами)
  • Проверка качества кода (review, тесты, прод, безопасность)
  • Проверка процесса (дейли, сроки, коммуникация)
  • Оценка и комментарии Tech Lead (что хорошо, что улучшить)
  • Расчёт оплаты (базовая ставка × процент приёмки, бонусы, штрафы)
  • Доработки (если приёмка с замечаниями)
  • Решение и подписи (TL, разработчик, PM, Owner, Бухгалтерия)
💡 Как пользоваться: жми «Сделать копию» — Google Docs создаст копию шаблона на твоём диске. Переименуй её под конкретную задачу/разработчика и заполняй. Оригинал шаблона остаётся нетронутым.
